# PM2: The Production Process Guardian

In 2026, while many developers use Docker for environment isolation,

PM2 remains the gold standard for Process Management
on single Linux servers (like your Arch setup) or within those containers.

If you run node app.js,
the moment you close your terminal or your code hits an error,
the server dies.

PM2 ensures your application stays alive 24/7,
handles crashes automatically,
and scales your app to use all your CPU cores

---

## 1. Why PM2 is Essential

Think of PM2 as a "bodyguard" for your Node.js process.

- Daemonization:
  It runs your app in the background.
  You can log out of your Arch machine,
  and the app keeps running.

- Auto-Restart:
  If your app crashes due to a memory leak or an unhandled exception,
  PM2 restarts it in milliseconds.

- Cluster Mode:
  Node.js is single-threaded.
  PM2 can "fork" your app into multiple instances
  (one for each CPU core)
  to multiply your performance.

- Startup Persistence:
  It can generate a script so that when your server reboots,
  your apps start up automatically.

---

## 2. Essential Commands for your Wiki

| Command                          | Purpose                                                     |
| -------------------------------- | ----------------------------------------------------------- |
| pm2 start app.js --name "my-api" | Start an app and give it a recognizable name.               |
| pm2 list (or pm2 ls)             | See a real-time table of all running apps and their status. |
| pm2 monit                        | Launch a terminal dashboard showing CPU and RAM usage.      |
| pm2 logs                         | Stream all your logs (standard and error) to the terminal.  |
| pm2 restart <id/name>            | Kill and restart the process.                               |
| pm2 reload <id/name>             | Zero-downtime restart (restarts instances one by one).      |

---

## 3. Scaling with Cluster Mode

On your Arch machine,
if you have a 4-core CPU,
your Node app is only using 25% of the hardware potential by default.
PM2 fixes this:

```bash
# Use 'max' to automatically detect and use all available CPU cores
pm2 start app.js -i max
```

## 4. Comparison: Nodemon vs. PM2 vs. Docker

| Feature      | Nodemon                                    | PM2                          | Docker                         |
| ------------ | ------------------------------------------ | ---------------------------- | ------------------------------ |
| Main Goal    | Auto-restart on code change.               | Process uptime & scaling.    | Environment isolation.         |
| Environment  | Development only.                          | Production (VPS/Bare Metal). | Deployment/Cloud.              |
| Clustering   | No.                                        | Yes (Built-in).              | Via Orchestrator (K8s).        |
| Arch Analogy | A script that runs ls when a file changes. | A systemd service manager.   | A complete virtual disk image. |

---

## 5. The "Set and Forget" Setup

To make sure your apps survive a server reboot on Arch Linux:

- Start your apps:
  pm2 start app.js

- Generate startup script:
  pm2 startup (Follow the command it prints!).

- Freeze the list:
  pm2 save (This saves the current list to be restored on boot).
